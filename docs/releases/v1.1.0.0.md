# v1.1.0.0

## What changed

- **Path validation for bridge methods** — All local file paths received from the JavaScript frontend are now validated before file I/O. The `ValidateLocalPath` helper rejects null bytes, UNC/device paths (`\\server`, `\\?\`, `\\.\`), and path traversal sequences (`..`). `ListLocalDirectory` is scoped to user profile directories (Documents, Desktop, Downloads); `Upload`/`Download` accept any normalized absolute path.
- **WebView2 message origin validation** — `OnWebMessageReceived` now validates that incoming messages originate from the expected virtual host (`ftpclient.local`) before processing. Messages from unknown origins are rejected with a warning logged to stderr.
- **SFTP host key first-connection notification** — On first SFTP connection (TOFU), the host fingerprint is now posted to the frontend via a `hostKeyTrusted` message so the user can see and verify the key being trusted.
- **SaveSite error feedback** — The previously silent catch-all in `SaveSite` now posts a `siteSaveError` message to the frontend when credential persistence fails, so the user knows the save did not persist.
- **Error message sanitization** — A `SanitizeErrorMessage` helper strips IPv4 addresses and Windows absolute paths from exception messages before they are forwarded to the JavaScript frontend, preventing internal details from leaking to the UI.

## Why

These changes address confirmed findings from the PR #1 code review. The security hardening (Phase 1) closes path injection, origin spoofing, and blind TOFU trust vectors. The UX improvements (Phase 2) provide user-visible error feedback where errors were previously silent.

## Migration steps

None — all changes are backward-compatible. The frontend should handle the new message types (`hostKeyTrusted`, `siteSaveError`) gracefully.

## Known caveats

- `ListLocalDirectory` path validation uses English folder names ("Downloads", "Documents") which may not match localized folder names on non-English Windows systems.
- Path validation rejects filenames containing ".." as a substring (e.g., `file..txt`) — overly restrictive but safe.
- Error sanitization only applies to `PostError` call sites; `PostTransferEvent` and `ListLocalDirectory` error paths are not yet sanitized.
